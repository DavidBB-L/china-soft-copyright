# 中国软著申请材料生成

为中国软件著作权登记生成合规的申请材料。

## 文档要求速查

| 材料 | 页数规则 | 每页行数 | 格式 |
|------|---------|---------|------|
| 程序鉴别材料 | 源程序>60页→前30+后30页；<60页→全部 | ≥50行/页 | 行号+代码，等宽字体 |
| 文档鉴别材料 | 全文档>60页→前30+后30页；<60页→全部 | ≥30行/页 | 说明书/用户手册/设计文档 |

## 申请引导流程（必读）

当用户说"我要申请软著"或"帮我准备软著材料"时，**先走完下面的引导问答，再动手生成材料**。不要跳过直接生成——填错信息比没材料更麻烦。

### Step 1: 基本信息采集

依次问用户以下问题（一次问不完分几次，但每个都要确认）：

**Q1: 软件全称是什么？**
- 提示：名称里用最宽泛的领域词，覆盖面越大越好
- 禁止用型号/版本号当名称（如"V1.0"不行）
- 须以"软件/APP/系统/平台"结尾
- 一般15字以内
- 正确示例：`XX内容创作工具软件`、`XX智能写作平台`

**Q2: 英文名要不要加？**
- 建议加，显得正式。格式：`中文名 英文名 软件类型`
- 示例：`XX智能写作平台 XX SmartWrite Platform`

**Q3: 版本号？**
- V1.0 是最常见的首次申请版本
- 如果已有旧版软著，新版本用 V1.1、V1.2 等

**Q4: 著作权人是谁？**
- 自然人：需要身份证上的姓名
- 公司：需要营业执照上的全称
- 多个著作权人：需要各自的证件信息

**Q5: 开发方式？**
- 独立开发（最常见，一个人做的）
- 合作开发（多人/多公司合作）
- 委托开发（甲方委托乙方）
- 下达任务开发（上级下达给下级）

**Q6: 权利范围？**
- 全部权利（最常见）
- 部分权利（需要具体说明哪些权利）

**Q7: 首次发表？**
- 未发表（首次申请最常见，选这个）
- 已发表（需要填发表日期）

### 字数限制速查表

| 字段 | 字数限制 | 备注 |
|------|---------|------|
| 软件全称 | 一般15字以内 | 须以"软件/APP/系统/平台"结尾 |
| 软件简称 | 不超过全称长度 | 可不填 |
| 版本号 | 无限制 | 格式：V1.0 或 1.0 |
| 源程序量 | 无限制 | 填精确行数 |
| 开发目的 | 无明确限制 | 1-2句话即可 |
| 面向领域/行业 | 无明确限制 | 简短描述 |
| 主要功能 | **500-1300字** | 需与说明书一致，审核重点 |
| 技术特点 | **≤100字** | 必须精炼，体现独创性 |
| 硬件环境 | 无明确限制 | 简短描述 |
| 软件环境 | 无明确限制 | 简短描述 |
| 编程语言 | 无明确限制 | 列出所有语言 |

⚠️ **2026年3月15日更新**：中国版权保护中心更新了申请表，功能说明部分有扩展，签章环节新增诚信说明要求。旧版申请表已废止。

### Step 2: 技术信息采集（有字数限制）

**Q8: 源程序量（总行数）？**
- 告诉用户：先统计代码行数，判断是"全部提交"还是"前30页+后30页"
- 规则：代码量≥3000行 → 前30+后30（共60页）；<3000行 → 全部
- 帮用户跑 `wc -l` 统计

**Q9: 软件分类？**
- 常见分类：
  - `30200-0000` — 应用软件（最通用）
  - `30100-0000` — 系统软件
  - `30400-0000` — 嵌入式软件
- 如果用户不确定，默认选 `30200-0000`

**Q10: 开发完成日期？**
- 必须早于申请日
- 建议填实际开发完成时间，与代码提交记录（Git日志）一致

**Q11: 开发硬件环境（≤50字）？**
- 示例：`Intel Core i7 CPU, 16GB内存, 512GB固态硬盘`

**Q12: 运行硬件环境（≤50字）？**
- 示例：`Intel i3 及以上CPU, 2GB内存, 100MB磁盘空间`

**Q13: 开发软件的操作系统（≤50字）？**
- 示例：`Windows 10及以上, macOS 12及以上`

**Q14: 软件开发工具/环境（≤50字）？**
- 示例：`VS Code, Node.js v18, Electron 33, Git`
- 注意：开发工具不能当编程语言填

**Q15: 运行平台/操作系统（≤50字）？**
- 示例：`Windows 10/macOS 12/Linux, Chrome/Firefox/Safari`

**Q16: 运行支撑环境/支持软件（≤50字）？**
- 示例：`Node.js v18及以上, Chromium浏览器内核`

**Q17: 编程语言？**
- 无明确字数限制，列出所有使用的语言
- 示例：`JavaScript, HTML, CSS, JSON`
- 多种语言都要写全

**Q18: 面向领域/行业（≤50字）？**
- 示例：`根据实际应用领域填写，如：教育、医疗、金融、电商、内容创作等`

**Q19: 开发目的（≤50字）？**
- 一句话说明为什么做这个软件
- 示例：`解决XX行业的XX问题，提高XX效率`

**Q20: 主要功能（500-1300字）？**
- ⚠️ 这是最重要的字段，字数区间 500-1300 字
- 着重描述软件的各项主要功能，并简述特点
- 内容必须与说明书一致
- 引导用户按功能模块逐个描述

**Q21: 技术特点（≤100字）？**
- ⚠️ 只有100字，必须精炼
- 写明软件实现的功能或改进，体现独创性
- 示例：`基于AI大模型的智能写作工具，内置多套行业模板，支持智能生成、多维度改写、实时校验。采用跨平台架构，支持桌面端及移动端。`

**Q22: 其他证明文件？**
- 根据情况可能需要：
  - 合作开发协议（多人合作时）
  - 原著作权许可证明（基于他人软件开发时）
  - 权利继承/转让证明（继承或受让情况）
  - 非职务软件开发保证书（著作权人为自然人时）
  - 前期开发说明（开发完成时间与企业成立时间相近时）

### Step 3: 材料生成

信息确认完毕后，按下面顺序生成：

1. **源代码文档** (.txt) — 50行/页，行号+代码
2. **软件说明书** (PDF) — ≥30行/页，含功能介绍+截图
3. **申请表填写指导** — 列出每个字段的正确填写方式

### 常见坑（提前告诉用户）

- **软件全称不能改**：提交后改名等于重新申请
- **版本号不影响保护范围**：V1.1 的软著覆盖后续所有版本
- **重构不影响软著**：改内部代码不改变软件名称+功能形态，证书继续有效
- **源代码文档里的软件名必须跟申请表一致**：差一个字都不行
- **命名选宽泛词**：领域词选覆盖面大的，别用太窄的把自己锁死（如"剧本"比"短剧"好，"内容创作"比"短视频"好）
- **2026年3月15日起使用新版申请表**：旧版已废止，新版新增诚信声明要求
- **主要功能字段（500-1300字）是审核重点**：必须与说明书内容一致
- **技术特点只有100字**：提前精炼好，别写多了被截断
- **各50字字段别超限**：表单会直接截断，超了不提示
- **软件全称须以"软件/APP/系统/平台"结尾**：以"工具/计算/系列"等结尾的要慎用

## 工作流程

### 1. 确认源程序量
```bash
cat *.js *.html *.css | wc -l
```
算出总行数，判断是前30+后30还是全交。

### 2. 生成源代码文档 (.txt)
用 `scripts/gen_source_doc.py` 生成精确 50 行/页的源代码 .txt 文件。
- 包含行号前缀 `    1 | code...`
- 页间用 `--- 第 N 页结束 ---` 标记
- 头尾包含软件名称、版本号、著作权人、总行数

### 3. 生成软件说明书
用 `scripts/gen_manual_pdf.py` 从 .md 生成 PDF。
- 字号 9.8px / 行高 1.55 / 边距 1.3cm 左右 → 保证每页 ≥30 行
- 如果末页不足 30 行，在说明书里补充版本历史/测试报告等正经内容把它填满

### 4. PDF 生成避坑
**不要用 Chromium CLI 的 `--print-to-pdf`**，它常常忽略 `@page` CSS 导致行数不可控。
**不要用 Python Playwright**（环境不一定有）。

正确方式：**Node.js Playwright API**，margin 通过 API 传，CSS 里不写 `@page`：

```javascript
const { chromium } = require('playwright');
const browser = await chromium.launch({ headless: true });
const page = await browser.newPage();
await page.goto('file://' + htmlPath, { waitUntil: 'networkidle' });
await page.pdf({
  path: outputPath,
  format: 'A4',
  margin: { top: '0.5cm', bottom: '0.5cm', left: '0.5cm', right: '0.5cm' }
});
```

关键参数：
- 源码 PDF：`font-size: 8.5px; line-height: 1.18; margin: 0.5cm` → 50 行/页
- 说明书 PDF：`font-size: 9.8px; line-height: 1.55; margin: 1.3cm` → 30+ 行/页

**验证每页行数：**
```bash
pdftotext output.pdf - -f $PAGE -l $PAGE | grep -c '|'  # 源码
pdftotext output.pdf - -f $PAGE -l $PAGE | grep -c .    # 说明书
```

## 命名策略
软件全称中的领域词选最宽泛的那个——覆盖面越大越好，不要用太窄的词把自己锁死。例如"内容创作"比"短视频"好，"写作工具"比"剧本编辑器"好。名称须以"软件/APP/系统/平台"结尾，以"工具/计算/系列"等结尾的要慎用。

## 支撑文件
- `scripts/gen_source_doc.py` — 源代码 .txt 生成器（50行/页）
- `scripts/gen_manual_pdf.py` — 说明书 MD→PDF 转换器（≥30行/页）

## PDF 生成避坑

### 行数验证
```bash
# 源码PDF：统计含 '|' 的行（行号前缀）
pdftotext output.pdf - -f $PAGE -l $PAGE | grep -c '|'
# 说明书PDF：统计非空行
pdftotext output.pdf - -f $PAGE -l $PAGE | grep -c .
```
⚠️ pdftotext 可能比肉眼少计1-2行，以实际 PDF 页面目视为准。

### 48行诅咒
字体调再小行数也不变时，检查是否 `@page` margin 和 Playwright API margin 叠加了。**不要用 `@page` CSS** — 全部 margin 通过 Playwright API 传。

### 源码 PDF 参数（精确50行/页）
- `font-size: 8.5px; line-height: 1.18; font-family: "Courier New"`
- margin: `0.5cm` 四边
- 每50行一个 `<pre>` 块，块间 `page-break-before: always`

### 说明书 PDF 参数（≥30行/页）
- `font-size: 9.8px; line-height: 1.55`
- margin: `1.3cm` 四边
- Markdown → HTML 用 `mistune` 或手写转换

以上参数经过多次调试，不要随意改。
