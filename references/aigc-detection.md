# AIGC 痕迹检测指南

## 检测流程

**不要预设检测清单。** 让AI用自己的专业知识从文本整体风格、结构模式、用词习惯等维度综合判断。预设清单会限制检测范围——你不知道AI会从哪些角度识别AI文本，所以不要替它决定。

**推荐提示词（给检测AI）：**
```
你是AIGC检测专家。请读取以下材料，用你自己的专业知识判断每份材料是否包含AI生成痕迹。

材料路径：[路径]
- 文件1
- 文件2
- ...

逐份分析，输出：
- 文件名
- 风险等级：🟢安全 / 🟡低风险 / 🔴高风险
- 发现的问题及具体位置
- 修改建议

不要只查关键词，要从文本整体风格、结构模式、用词习惯等维度综合判断。
```

## 已知AIGC痕迹模式（2026-06实测）

以下模式由检测AI实际发现，不是预设清单——是真实案例积累：

### 1. 品牌名泄露（不只是AI模型名）

不只是GPT/Claude，还包括AI工具生态的品牌：
- **LM Studio**、**Ollama**（本地推理工具）
- **OpenAI**（API兼容接口描述）
- 各种云服务商名称

**修复**：改成通用描述
- "支持本地LM Studio" → "支持本地推理服务"
- "OpenAI兼容接口" → "云端API兼容接口"
- 配置示例中 `lmstudio`/`openai`/`ollama` → `local`/`openai-compat`/`local-oss`

### 2. AI自检笔记（meta-commentary）

AI生成申请表填写指南时，容易在"注意事项"段留下自检汇报：
- "不含任何AI模型名称（GPT/Claude等）" ← 把模型名写出来了
- "字数约800字" ← AI在统计自己的输出
- "不带字数标注" ← 这句话本身就是在标注字数
- "内容与说明书一致" ← AI在做一致性校验
- "填2026-06-01" ← "填"是AI对编写者的指令

**判断标准**：注意事项的内容是"已完成的汇报"还是"待完成的提醒"？前者是AI痕迹，后者是人类写法。

**修复**：删掉整个注意事项段，或改成开发者自己的checklist语气。

### 3. 句式同质化

AI把说明书浓缩成申请表时，每个句子的骨架趋于一致：
- "支持XX，可从XX导入，支持XX，每个XX有XX"
- 每个功能模块的描述篇幅几乎相同
- 嵌套括号 + "此外还有"收尾 + 逗号分隔的平行项

**人类特征**：挑重点写详细，次要功能一笔带过，篇幅分配不均匀。

**修复**：挑4-5个最核心的功能用自然句子写，允许次要功能被省略。

### 4. 技术特点的平行压缩

AI被要求"≤100字"时，会把每个技术点压缩成5-9字短语用顿号串联：
> "纯XXX原生架构，XXX流式返回，XXX防数据损坏，XXX动态注入，XXX双主题，XXX双端布局，XXX拆解长文本"

**修复**：拆成2-3句有主谓宾的完整句子。

### 5. 字数标注

人类写东西不会在后面标字数。"(57字)"、"（共XXX字）"是AI生成文本的经典标志。

### 6. 过度对仗的测试报告

通篇"支持XX，支持XX，支持XX"三连、"0 Bug，0 JS错误"、"P0×7 + P1×9"这类项目管理编号格式。

**修复**：改成开发者回忆测试过程的叙述体。

### 7. 说明书与申请表风格不一致

如果两份文档风格差异太大（说明书像人写的、申请表像AI写的），审查员可能起疑。

**修复**：申请表指南重写后，确保风格和说明书有重叠——保留一些口语化短句、允许信息密度不均匀。

### 8. 截图描述的句式同质化

为每张截图写描述时，AI容易用完全相同的句式开头：
- "上图展示了XXX的实际效果" — 8个功能模块8次同样的开头
- 每段都是100-200字，信息密度完全一致
- 描述结构雷同：功能说明→截图→上图展示了→详细描述

**人类特征**：有的截图写详细、有的一句话带过、有的不写描述直接让图说话。

**修复**：每段用不同句式开头，长短不一。重要的2-3段写详细，其余一两句话。

### 9. AI填充虚数

AI在处理语义时，会自动填充一个听起来合理的数字：
- "导出测了六种格式都能正常生成" → 实际只支持4种
- "支持七种关系类型" → 实际可能只有5种

**修复**：所有数字必须和代码/实际功能核对。不确定就写具体名称不写数字。

### 10. 编造测试条目

AI加的测试条目往往缺乏具体场景，读起来像"应该测了什么"而非"实际测了什么"：
- "SSE流式返回在弱网环境下能正常降级" — 什么弱网？用什么工具模拟的？
- "原子写入在模拟断电场景下数据完整性验证通过" — 怎么模拟的断电？

**判断标准**：人类写的测试报告通常带工具名/环境/操作步骤，AI写的只有结论。

**修复**：要么加具体细节（"Chrome DevTools限速50KB/s下测试"、"kill -9进程后重启验证"），要么直接删掉。

## 申请表填写指南的AIGC检测（2026-06-06实测）

申请表填写指南（给用户看的字段填写参考）也需要过AIGC检测。AI写指南时有以下泄露模式：

### 模式A：自检汇报代替待办提醒
- ❌ AI写法：`"技术特点字段已确认不含GPT、Claude等，共76字"`
- ✅ 人类写法：`"提交前核对技术特点字段，控制在100字以内"`
- 判断标准：内容是"已完成的汇报"还是"待完成的提醒"？前者是AI痕迹

### 模式B：列举禁止项时把禁品名字写出来
- ❌ AI写法：`"不含GPT-4、Claude 3.5等AI模型名"`
- ✅ 人类写法：`"不含任何AI模型名称"`
- 问题：列举本身就是泄露。人类不会在注意事项里写一份AI模型清单

### 模式C：字数自报
- ❌ `"主要功能约800字"` / `"技术特点76字"` / `"（57字）"`
- ✅ 不提字数，或者写"控制在500-1300字区间"
- 人类不会精确统计自己写了多少字

### 模式D：字段值复制时带残留标注
申请表字段内容在复制粘贴时，可能带上工具生成的字数标注。**提交前逐字段肉眼检查**，确保没有：
- 末尾的`（XX字）`
- 开头的分类标签（如技术特点前面多了分类名）
- 括号内的AI注释
